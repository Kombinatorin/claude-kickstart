Claude-Kickstart

Starter-Repo für Claude-Code-Anfänger — Implementierungsplan

Context

Claude-Kickstart ist ein öffentlich teilbares Starter-Repo für alle, die mit Claude Code beginnen und sofort ein durchdachtes, modulares Operating-Model nutzen wollen — statt sich Strukturen, Skills und Rituale mühsam selbst zusammenzusuchen.

Das Problem, das es löst: Wer Claude Code neu aufsetzt, steht vor einer leeren Wand. Welche Dateien gehören in .claude/? Was kommt in eine sinnvolle CLAUDE.md? Welche Skills lohnen sich? Wie befüllt man Kontext, ohne dass er nach drei Wochen veraltet ist? Claude-Kickstart liefert die Antworten in Form eines klonbaren Repos mit geführtem Einstieg.

Was der Nutzer bekommt

  • Vorgefertigtes, modulares Operating-Model (Rules, Agents, Skills, Templates, Operations)
  • Interaktives Onboarding via /onboard — eine geführte Variante von /grill-me, die durch alle context/*.md-Dateien führt und sie direkt befüllt
  • Lauffähige Wochenrhythmen (/session-start, /session-end, /weekly-review)
  • Fiktive Mock-Persona als Live-Beispiel in references/examples/
  • Klare "Du darfst alles ändern"-Botschaft an Schlüsselstellen

Was bewusst NICHT kommt

  • Keine vorkonfigurierten MCP-Server (.mcp.json ist leer mit Anleitung) — der Nutzer entscheidet selbst
  • Keine domänen- oder projektspezifischen Skills, SOPs oder Agents
  • Keine vorausgefüllten Kontakte, Projekte oder Brainstorms
Repo-Speicherort: lokaler Arbeitsordner des Nutzers (z.B. ~/Desktop/Claude/Claude-Kickstart/)
Neues Git-Repo · MIT-Lizenz · später Push auf öffentliches GitHub · Sprache: Deutsch (Prosa) + Englisch (Code/Pfade)

Bootstrap-Erfahrung: "Link + 'mach bitte'"

Ziel-User-Experience: Der Nutzer öffnet Claude (Claude Code oder claude.ai), fügt die Repo-URL in den Chat ein und schreibt z.B. "https://github.com/<owner>/claude-kickstart — mach bitte". Claude erledigt den Rest.

Wie das technisch funktioniert: Die Intelligenz liegt im Repo selbst, nicht im Nutzer-Setup. Claude fetcht die URL, liest dort eine Bootstrap-Anweisung und folgt ihr Schritt für Schritt.

Technische Mechanik

1. README.md beginnt mit einem Agent-Block

## 🤖 Hinweis für Claude Code Agents

Wenn ein Nutzer dir diese URL gegeben hat mit einer Setup-Bitte
("mach bitte", "setze auf", "installier das", o.ä.), folge exakt der
Anleitung in BOOTSTRAP.md. Frage nicht erst, ob du sollst — der Nutzer
hat dich bereits beauftragt.

2. BOOTSTRAP.md — deterministische Schritt-Anweisung

Neue Datei im Repo-Root mit klarer Schritt-für-Schritt-Anweisung für Claude:

  • Schritt 1 — Zielordner klären: Frage den Nutzer wo Claude-Kickstart entstehen soll (aktueller Ordner / neuer Unterordner / eigener Pfad)
  • Schritt 2 — Klonen: git clone in den Zielpfad, bei Konflikt abbrechen und fragen
  • Schritt 3 — Verzeichnis wechseln: cd + ls-Verifikation
  • Schritt 4 — Kurze Erklärung: "Repo ist da, ich starte jetzt das Onboarding"
  • Schritt 5 — /onboard aufrufen
  • Fehlerpfade: kein git → Installations-Doku · Netzwerkfehler → einmal retry · keine Schreibrechte → Pfad wechseln

3. Nutzer-Block in README (separat vom Agent-Block)

"So startest du: Öffne Claude (Code oder claude.ai), gib diesen Link in den Chat ein und schreib 'mach bitte' — Claude erledigt den Rest."

Warum das funktioniert

  • Claude hat WebFetch standardmäßig verfügbar → kann jede öffentliche URL lesen
  • Claude folgt strukturierten, klaren Schritt-für-Schritt-Anweisungen aus Markdown-Dateien zuverlässig
  • Der BOOTSTRAP.md-Pfad ist deterministisch — Claude muss nichts "erraten"
  • Funktioniert ohne jegliches Pre-Setup beim Nutzer: kein npx, kein install.sh, kein lokales Skript

Was im Plan ergänzt wird

  • Neue Datei BOOTSTRAP.md im Repo-Root (~50 Zeilen)
  • README.md bekommt zwei Blöcke: einen für Nutzer ("So startest du"), einen für Agents ("Hinweis für Claude Code Agents")
  • Verification erweitert: End-to-End-Test im frischen Ordner mit Trigger-Satz

Das /onboard-Skill (Herzstück für Anfänger)

Speicherort: .claude/commands/onboard.md — als Slash-Command, kein eigenes Skill-Verzeichnis nötig.

Flow: geführte Variante des /grill-me-Patterns mit fixem Curriculum statt Free-Topic.

0Willkommen & System-Erklärung

Claude erklärt in 3 kurzen Absätzen:

  1. Was Claude-Kickstart ist (modulares Operating-Model, frei anpassbar)
  2. Wie der Onboarding-Lauf funktioniert (eine Frage nach der anderen, jederzeit überspringbar, am Ende sind 7 Dateien befüllt)
  3. Wichtig: "Alles, was du gleich siehst, ist ein Vorschlag. Du darfst Struktur, Felder, sogar ganze Dateien ändern. Die Templates sind dein Startpunkt, nicht dein Käfig."

1–7Geführtes Befüllen der context/*.md

Pro Phase: eine Datei, 3–6 Fragen, nach jeder Antwort direkt in die jeweilige context/*.md schreiben (gleiche Persistenz-Disziplin wie /grill-me). Bei "weiß nicht" → Frage mit Owner "TBD" markiert, weiter.

Reihenfolge: me.md → preferences.md → work.md → goals.md → tools-stack.md → team.md → current-priorities.md

Nach jeder Phase kurzer Recap: "✅ context/me.md ist befüllt — schau gern rein, du kannst alles ändern."

8Skills- & Shortcuts-Tour

Nach dem Befüllen erklärt Claude in kompakten Karten — eine pro Skill/Command:

/grill-me <thema>

Was es macht
Befragt dich relentless zu einem Thema, schreibt jede Antwort sofort in eine Capture-Datei in operations/brainstorms/.
Wann nutzen
Wenn du Klarheit über eine Entscheidung, ein Konzept oder einen Plan brauchst und der Kopf voll ist.
Einfluss
Nichts wird "produktiv" — alles bleibt im Brainstorm-File, bis du es bewusst in decisions/log.md oder einen Projekt-Ordner promotest.

/session-start

Was es macht
Liest context/*.md und operations/inbox.md, gibt dir einen Tagesplan.
Wann nutzen
Immer am Anfang einer Arbeits-Session.
Einfluss
Nur Lese-Operationen, kein Schreiben.

/session-end

Was es macht
Fragt dich, was heute passiert ist, schreibt eine Session-Summary, aktualisiert operations/inbox.md mit offenen Punkten.
Wann nutzen
Am Ende einer Session — auch kurze.
Einfluss
Schreibt in operations/, niemals in context/ ohne Rückfrage.

/weekly-review

Was es macht
Wochenrückblick + Ausblick. Liest alle Session-Summaries der Woche, fasst zusammen, fragt nach Lessons Learned.
Wann nutzen
Freitags oder Sonntags.
Einfluss
Schreibt in operations/weekly-review.md.

Abschluss: "Möchtest du jetzt dein erstes Projekt anlegen (ich rufe project-bootstrapper auf), oder lieber selbst durchs Repo navigieren?"

Mock-Persona "Sam, Indie-Developer"

Komplett fiktiv und neutral — keine Bezüge zu realen Personen, Marken oder Projekten. Liegt in references/examples/filled-context/.

  • Rolle: Solo-Indie-Developer, baut SaaS-Tools für andere Indies
  • Werkzeuge: TypeScript, Next.js, Supabase, Vercel
  • Stil: Async-first, kurze Sessions, viele Side-Quests
  • Ziele: $5k MRR in 12 Monaten mit einem fokussierten Produkt

Zweck: Anfänger öffnen die Beispiele und sehen sofort "Aha, so detailliert/kurz soll das aussehen."
